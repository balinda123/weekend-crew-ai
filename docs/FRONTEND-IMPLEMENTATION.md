# 《馋猫局儿》前端实施说明

**负责人：** AI 应用前端 / 全栈
**目标：** 用 Taro + React + TypeScript 实现一个可真实组局、可解释 AI 结果的小程序体验。

本说明依赖 [共享技术设计](TECHNICAL-DESIGN.md)。接口字段、权限和错误码以 OpenAPI 为准，本文件不复制后端规则。

## 1. 前端技术基线

| 范围 | 选择 |
|---|---|
| 框架 | Taro + React + TypeScript（`strict: true`） |
| 基础组件 | NutUI React Taro |
| 样式 | SCSS + CSS Variables；业务组件自建 |
| 服务端状态 | TanStack Query 或等价请求缓存层 |
| 轻量 UI 状态 | Zustand 或 React Context，避免全局状态堆叠 |
| 表单校验 | Zod，与 API DTO 类型分开维护 UI 表单模型 |
| 地图 | 小程序地图能力 + 后端返回的高德业务数据；不在客户端暴露高德 Web 服务 Key |
| 测试 | 纯逻辑单测 + 核心流程端到端测试 |

TypeScript 是必须项：它让自动生成的 API 类型、地图状态、账本分摊编辑和 AI 结果渲染在编译期暴露字段不一致。

## 2. 页面与路由

| 页面 | 主要内容 | 关键交互 |
|---|---|---|
| 首页/我的局 | 局列表、创建局、历史局 | 创建、加入、归档。 |
| 局概览 | 日期、成员、进度、下一步 | 入口汇总，不承载全部功能。 |
| 地图与地点池 | 地图、地点卡片、投票、状态筛选 | 列表/地图双向定位；芝士提示下一站。 |
| 添加地点 | 手动搜索、来源链接、截图上传 | OCR 候选必须人工确认。 |
| AI 行程 | 约束输入、2–3 套方案、解释、拖拽编辑 | 展示“数据不足/生成失败/可重试”。 |
| 现场打卡 | 当前行程、已去/跳过、快捷记账 | 打卡不自动修改历史行程。 |
| 账本 | 明细、付款人、参与人、分摊方式、结算 | 奶糖展示结算成功；金额以分提交。 |
| 攻略预览/分享 | 脱敏字段选择、预览、复刻入口 | 显式勾选可公开内容。 |

## 3. 组件边界

- `MapCanvas`：仅负责地图视图、marker、polyline、事件转发；不直接请求业务 API。
- `PlaceSheet`：地点列表与筛选；选择地点时调用统一的 `selectedPlaceId`。
- `PlaceCard`：地点状态、来源、投票和操作入口；不可内嵌地图逻辑。
- `ItineraryPlanCard`：消费 API 返回的结构化计划，显示取舍原因；不得解析模型原始文本。
- `ExpenseSplitEditor`：编辑付款人/参与人/份额；本地先校验总额，再提交服务器。
- `SettlementPanel`：只呈现服务端确定性结算结果。
- `CatGuide`：芝士/奶糖的状态化展示组件。状态仅限 `idle | suggesting | loading | success | warning`，文案由业务场景传入，避免把吉祥物做成全局聊天框。

## 4. API 使用规则

1. 所有网络请求经过 `apiClient`；由 OpenAPI 生成的类型定义请求与响应。
2. 请求层负责 token、`request_id`、超时、统一错误映射；页面不手写 HTTP 错误分支。
3. Query key 必须包含 `crewId` 和资源版本；写操作成功后精准失效相关查询。
4. 只有 OCR/AI 等异步任务可轮询；轮询间隔递增且在页面卸载时取消。
5. AI 行程响应为未通过 schema 校验或状态非 `ready` 时，不渲染为正式行程。

## 5. 地图交互规范

- 默认只加载已确认地点；候选 POI 在确认页局部显示，不能污染正式地图。
- Marker 使用类别和状态颜色区分；芝士仅显示在“下一站/推荐路线起点”，奶糖不作为常驻地图 marker。
- 列表点击地点：地图聚焦并展开气泡；地图 marker 点击：选中对应列表卡片。
- 地图失败时自动回退为按距离排序的地点列表，保留投票、打卡与记账能力。
- 输入提示做 300–500ms 防抖，最少输入 2 个字符；不把每一次键入都变为 POI 请求。

## 6. AI 体验规范

- AI 入口使用任务导向文案，例如“按大家想吃的店排一天”，不做空泛的聊天首页。
- 明确展示输入来源：人数、日期、出发地、已确认地点、投票和未满足约束。
- 生成过程中显示可取消进度；网络错误、配额耗尽、数据不足分别给出下一步操作。
- 方案卡必须显示：地点顺序、预计时段、交通提示、取舍原因、未采用地点。
- 用户拖拽后显示“已手动调整”；再次“重新排”只基于当前版本，不丢失旧方案。

## 7. 前端质量门槛

- 禁止 `any` 和未类型化的 API 响应；关键业务状态使用可辨识联合类型。
- 账本编辑、行程拖拽、图片上传和分享字段选择均有表单校验与失败提示。
- 覆盖至少四条端到端流：创建局并加入、手动加地点、记账后结算、生成并复刻私密攻略。
- 视觉检查重点：小屏设备、弱网、无地图/无 AI、长地点名、多成员头像和金额过长。

## 8. 你的第一阶段交付

1. 初始化 `apps/miniapp`，配置 TypeScript、NutUI、请求层和 mock API。
2. 做“创建局 → 地点池 → 地图/列表联动”的可点击主路径。
3. 与后端共同冻结第一批 OpenAPI：组局、成员、地点、POI 候选、账本。
4. 做账本分摊编辑器和结算结果页；以 API 返回为唯一计算真相。
5. 接入 AI 方案卡、任务状态和芝士/奶糖状态组件。
